Querying the Initial Password for Logging In to the VM

If the VM is created on FusionCompute, and the password is random, you can query the initial password for logging in to the VM.

Choose Resource > Resource Clusters, select a resource cluster, navigate to the VMs page, select a VM, navigate to the Summary page, click More, and select Query Initial Password.

Logging In to a VM Using VNC

VNC is a remote terminal access tool. You can use VNC to remotely connect to a VM, view the VM operating page, and perform management and maintenance operations on it.

However, if the VM operating system (OS) requires password input for login, you need to enter the correct OS username and password to access the VM operating page.

Note the following points when you log in to a VM using VNC:
  • Declaration: This feature is a high-risk feature. Using this feature complies with industry practices. However, end user data may be required for implementing the feature. Exercise caution and obtain end user's consent when using this feature.This operation involves user privacy issues. Therefore, ensure that your login complies with the local laws.

  • To ensure VM security, the VM user must change the VM login password in the earliest time.

  • If you, as an administrator, have logged in to a VM, the VM service VM will not be allowed to log in to the VM before you log out.

If any exception occurs during login, address the issue based on the method described in VNC Login Troubleshooting.

VM Power Management

Power management for a VM includes hibernate, start or wake up, stop, restart the VM. Stop or restart a VM can be performed in a safe or forcible manner. You are advised to use the safe manner. Forcibly stop or restart a VM only if the VM cannot be safely stopped or restarted due to an OS fault. Forcibly stopping or restarting a VM may damage the VM disks and cause data loss.

Starting or stopping a large number of VMs increases host workload. If you need to start or stop a large number of VMs on a host, perform the start or stop operations in batches to prevent VM services on the host from being adversely affected.

Choose Resource > Resource Clusters, select a resource cluster, navigate to the VMs page, select a VM, navigate to the Summary page and select required operations. Some operations are collapsed in More.

Migrating a VM

VM migration refers to the process that a running VM is moved from one host to another host without service interruption.

If the service load is over light or heavy on the host where the VMs run or the host is faulty, migrate the VMs to ensure that they work properly.
  • If the service load is too heavy on the host or the host is faulty, migrate the VMs to another host to prevent service interruption due to a host exception.
  • If the service load is too light on multiple hosts, migrate the VMs to a small number of hosts to consolidate host resources and then power off the idle hosts. This improves resource utilization and reduces power consumption.

For details about the requirements, see VM Migration Requirements.

Choose Resource > Resource Clusters, select a resource cluster, navigate to the VMs page, select a VM, navigate to the Summary page, click More, and select Migrate.

Cloning a VM

Clone an existing VM to create a VM.

NOTE:

Declaration: This feature is a high-risk feature. Using this feature complies with industry practices. However, end user data may be required for implementing the feature. Exercise caution and obtain end user's consent when using this feature.

A VM clone inherits the configurations and data of its parent VM at the cloning time point. If the end user data is involved, obtain end user's consent before you clone the VM.

You can clone a VM only when the VM is in the Running or Stopped state. The following requirements are met if you want to make a VM clone from a running VM:
  • FusionStorage is not used on the VM disk.
  • Disks on the VM use virtualized storage and are not shared and set to be unaffected by snapshots.
  • The Tools has been installed and is running properly on the VM

Choose Resource > Resource Clusters, select a resource cluster, navigate to the VMs page, select a VM, navigate to the Summary page, click More, and select Clone VM.

Converting a VM into a Template

Convert a common VM into a VM template, which can be used to create other VMs.

The following requirements are met if you want to convert a VM to a template:
  • The VM is in the Stopped state.
  • No shared disk is attached to the VM.
  • The VM does not have snapshots.

Choose Resource > Resource Clusters, select a resource cluster, navigate to the VMs page, select a VM, navigate to the Summary page, click More, and select Convert to Template.

Adding a VM to FusionManager for Management

Add a VM created in the in-use hypervisor to FusionManager so that FusionManager can manage this VM, for example, monitor alarms of the VM or assign the VM to a VDC. After the VM is added to FusionManager, the network information about the VM configured in the hypervisor still cannot be managed by FusionManager.

Choose Resource > Resource Clusters, select a resource cluster, navigate to the VMs page, select a VM, navigate to the Summary page, click More, and select Incorporate.

Assigning a VM

You can assign a VM to the VDC administrator of a specific VPC. However, VM assignment may cause unavailability of the VM NC. Therefore, after assigning a VM, inform the VDC administrator of allocating the VM to a user and modifying the VM network attributes in a timely manner.

Choose Resource > Resource Clusters, select a resource cluster, navigate to the VMs page, select a VM, navigate to the Summary page, click More, and select Allocate.

Exporting a VM as a Template

You can export a VM to a local directory and save it as a template so that the template can be used to create VMs in other systems.

NOTE:

Declaration: This feature is a high-risk feature. Using this feature complies with industry practices. However, end user data may be required for implementing the feature. Exercise caution and obtain end user's consent when using this feature.

After the VM template is exported, the template can be imported into the system again and used to create a VM that inherits all its data. If the end user data is involved, obtain end user's consent before you export the VM template.

You can export a VM and save it as a template only when the VM is in the Running or Stopped state. If the VM is in the Running state, the VM disks must be created on virtualized data storage, including virtualized local disks, virtualized SAN devices, and NAS devices.

Choose Resource > Resource Clusters, select a resource cluster, navigate to the VMs page, select a VM, navigate to the Summary page, click More, and select Export VM.

A VM template consists of one OVF file and multiple VHD files.
  • OVF file: provides the description information about the VM. The file name is the same as the VM template, for example, template01.ovf.
  • VHD file: generated as a system file for each disk. The file name format is template name-xvddrive letter.vhd, for example, template01-xvda.vhd.

Restoring a VM

You can perform this operation when a running VM is faulty, and it cannot be logged in to using VNC. During the restoration, the system creates a VM using the template and attach the system disk of the created VM to the faulty VM. The original system disk of the faulty VM is attached to the created VM as a user disk. Data in the original system disk will not lose.

For details, see Restoring a VM.

Choose Resource > Resource Clusters, select a resource cluster, navigate to the VMs page, select a VM, navigate to the Summary page, click More, and select Repair.
